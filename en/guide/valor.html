# Valor / Securities Accounting

The icons for the program groups and programs of Valor in the launcher use the color blue to make them easily recognizable as belonging to Valor.

# Master Data

# Manage Clients

Clients in Moneysoft and thus in Valor are identified with numbers in the range 000 to 999. You manage the clients with the program Manage Clients in program group Clients. The most important parameters of each client are the following:

The national currency is important for cross rates: If there is an exchange rate needed for the base currency but unavailable a cross rate will be calculated using the national currency.

Different clients can work with different transaction sets i.e. sets of transaction codes. You can set it with the field Transaction set.

You can also decide on a client-to-client basis whether the FIFO method will be used for valuations, with the check box Calculate FIFO?.

You can specify a client language on the field Print language that will determine the language used for reports if you use the corresponding language selection mode Client language in a report program.

# Change Client

For each user separately, one of the existing Valor clients is the current client, and all Valor programs automatically use the client that is current at the time of the start of the program. The status bar shows you which client is the current one.

You can switch between clients with the program Change Client in program group Clients.

Important

This program (like its sister programs in Final and Emcal) switches the current client for Final, Valor and Emcal (if installed) simultaneously to keep them in sync. However, all programs that are already running i.e. have windows open do not change the client. To work with the new client you have to terminate and then restart those programs.

# Manage Refund Rates

The program Manage Refund Rates in program group Clients lets you define refund rates that will be used when recording transactions and when calculating deferred income. You start it directly from the program group Clients or from the program Manage Clients by using the button Manage Refund Rates.

You can define refund rates globally for a domicile, or for a domicile in connection with a certain client and/or a certain transaction code.

You can define the refundable and the non-refundable tax percentages.

Check the option Potentially refundable? if there is a double taxation agreement with the domicile and a potentially refundable tax at source; the program Calculate and Print Withholding Tax in program group Valuations will then mark a possible tax refund accordingly.

# Transaction Sets

Transaction sets are distinct sets of transaction codes as managed with the program Manage Transaction Definitions in program group Utilities. They exist to support multiple valuation methods like Expenses part of book value or Expenses in P/L.

In the program Manage Clients you can define which transaction set the client will use:

# Create Consolidated Client

The program Create Consolidated Client in program group Clients creates a consolidated client by basically copying everything from a number of separate clients into one big client. Later you can update such a consolidated client with the "delta" i.e. all changes that happened in the separate clients since the creation of the consolidated client or the last update.

Consolidated clients are the technical mechanism of Valor to allow easy and fast reporting over a number of clients which otherwise would be impossible because most Valor report programs are restricted to one client, the current one.

While it's technically possible to modify the data in a consolidated client this usually does not make sense. It's recommended to treat consolidated clients as read-only copies of the original clients that you also can delete and re-create as needed.

# Basic Data

# Manage Securities

Valor works with a single, global pool of securities that is common to all clients: All securities of all clients are defined here. You manage this pool with the program Manage Securities in program group Basic Data.

Securities number together with Postfix defines a unique main key for each security as known to Valor that you cannot change anymore once the security is created.

While it's possible to create new securities manually, it's recommended to use the apiD interface to SIX Financial Information ("Telekurs") whenever possible because the data about securities in the SIX database is of good quality, and it's simply faster.

A security has a status with the possible values Verified and Unverified. Newly created securities get a Unverified status to display clearly that someone with the necessary knowledge still needs to check and/or complete the data that came from the SIX database.

# Manage Client-Specific Data

The program Manage Client-Specific Data in program group Basic Data lets you configure a security as used within the context of a particular client. If needed you can go one step further and include the deposit in the context.

The most important data:

  • Base security (more commonly known as underlying)
  • Voucher currency (used as default when recording a transaction)
  • FIFO calculation (for this particular security instead of client-wide)
  • Asset category (used for reports)
  • Account assignment (accounts used when generating journal entries for FA)
  • Alternative stock exchange for getting valuation rates from SIX

# Manage Deposits

Deposits in Valor are mostly used to model safe custody accounts, with as many deposits per client as needed.

You manage deposits by pressing the F2 function key in program Manage Client-Specific Data while the cursor is on the field Deposit.

A new deposit is created by choosing and entering a new code plus a name at least in German and confirming with OK. Unfortunately the new deposit cannot be used right away, but only after a restart of the program.

# Manage Current Accounts

With the program Manage Current Accounts in program group Basic Data you can define up to 100 current accounts per currency or per currency and deposit. By setting the Type of transaction and/or Security type fields you can use different accounts depending on those criteria, e.g. using two separate current accounts for Buy and Sell transactions.

Current accounts are used for proposing account defaults when recording transactions and FECs.

# Manage Account Assignments

An account assignment is a list of FA accounts that play a role for a particular type of transaction, stored under a three-digit numerical code.

You create or modify an account assignment by pressing the F2 function key in program Manage Client-Specific Data while the cursor is on the field Account assignment. Unfortunately the new account assignment cannot be used right away, but only after a restart of the program.

You can work with sector code schemes (also known as industry classification schemes) in Valor, using codes with up to 3 levels of hierarchy main group / group / subgroup and optional external codes. You manage the schemes with the program Manage Sector Code Schemes in program group Basic Data:

# Assign Sector Codes

You tell which security belongs to which sector by using sector codes from your sector code schemes in the program Assign Sector Codes in program group Basic Data.

For a "normal" security i.e. one that is not a fund or something similar you assign single sector codes, at most one from each scheme that exists.

For funds or fund-like securities you can define of up to 10 components, each with sector code and percentage, in the frame Breakdown of target fund. Again, you can define as many such breakdowns as you have sector code schemes defined.

# Manage Exchange Rates

You manage exchange rates for all currencies that you work with in the program Manage Exchange Rates in program group Basic Data.

In principle, the system works as soon as there is a rate for each currency in regard to a single base currency (probably CHF) because Moneysoft is able to calculate cross exchange rates as needed.

An exchange rate can be valid:

  • Globally for all clients
  • Only for a certain client
  • Only for a certain deposit in a client

An exchange rate is valid starting from the given date up to the day before the next younger rate that is present.

# Currency Properties

For setting currency properties, or more exactly the properties of currency pairs, you use the dialog behind the Currency Properties button in program Manage Exchange Rates.

The properties Decimals and Rounding govern what happens with amounts after they are converted using an exchange rate. The property Unit of rate (German: Kotierung) tells Moneysoft for how many units of the currency the exchange rate is given.

You should specify properties for each occuring currency pair, also for pairs without stored rates that rely on the automatic calculation of cross rates.

In this dialog you also specify how spot and forward exchange rates for a given list of periods are imported from SIX through the apiD interface.

Important

If you work with two different "stock exchanges" i.e. rate data providers for spot and forward exchange rates make sure that they use identical base spot rates.

# Manage Valuation Rates

The program Manage Valuation Rates in program group Basic Data gives you access to the rates that Moneysoft uses to value securities.

A valuation rate can be valid:

  • Globally for all clients
  • Only for a certain client
  • Only for a certain deposit in a client

A valuation rate is valid starting from the given date up to the day before the next younger rate that is present.

# Manage Dividends

The dividends that you specify in program Manage Dividends in program group Basic Data with ex date, value date, currency and amount per unit are processed by the program Create Transactions for Earnings in program group Transactions. This results in prerecorded transactions as at ex date that you can update daily for changing exchange rates.

# Transactions

# Record Transactions

The program Record Transactions in program group Transactions is one the central programs of Valor:

The program works along the same principles as the Final program Create Journal Entries: You can freely change transactions here before posting them, referring to them using temporary IDs that vanish after posting.

Some details about the values for certain fields and their processing:

Security: Refer to a security using its ISIN. The security itself must exist already in order to use it here; if there is no client-specific data yet for the security in the current client it will be automatically created, using configured defaults e.g. for the account assignment. You check and modify that data in program Manage Client-Specific Data in program group Basic Data.

Transaction code: Use the appropriate code, e.g. 11 for a buy or 41 for a sale. Some other fields are open or closed for entering values depending on the particular transaction code used.

Date: Use the same date for Trade date and Voucher date. The voucher date will be used for the generated FA journal entries.

Settlement amount: Use net amounts in the currency of the current account used. The exchange rate of the settlement currency to the base currency of the current client will apply.

Description: The program generates a default transaction description depending on the transaction code given that you can modify as needed.

Rate curr. of issue: The program proposes the current exchange rate of the currency of issue of the security to the base currency of the client; modify as needed.

Rate of security: You must give the valuation rate in the currency of issue of the security which will be displayed to the left of this field. The result settlement amount times rate of security will be displayed to the right of the field.

Accrued interest: If the security is a bond enter the interest already accrued here.

Charges / Taxes: Use the fields in this frame to specify charges/taxes as needed, with currency (default will be the settlement currency), amount and exchange rate.

Financial accounting: This frame details the FA accounts used, according to the account assignment configured for the transaction code, with the corresponding amounts.

# Pre-Entries

Normally Valor only creates FA entries for Valor transactions when you actually post the transactions. As a new feature you can configure (for each client individually, if necessary) that FA entries are created earlier i.e. already when you record transactions.

The FA entries created this way are called pre-entries. Pre-entries technically are similar to unposted journal entries, and you can look at them with the same program Create Jounal Entries, but you can't modify them like normal unposted journal entries, and you cannot post them in Final.

The Valor program Record Transactions will keep corresponding pre-entries in sync with all changes you make, whether you record new transactions, modify or delete them.

If you post the transactions their pre-entries will be posted as well. After this those FA entries will be normal posted journal entries, with nothing indicating that they started as pre-entries.

Not only the program Record Transactions will create pre-entries and keep them in sync if configured to do so, but all other programs that somehow create transactions as well, e.g. the program Record Forward Exchange Contracts.

Many Final and Valor programs that generate reports have an option to include or exclude unposted journal entries. That same option now includes or excludes pre-entries as well.

# Cancel Transactions

You cancel transactions in the program Record Transactions in program group Transactions with the following process:

In the main dialog, leave field ID empty and fill the fields in frame Security information to specify the security with a transaction to cancel. Then click on the Cancel... button (top right corner of the dialog).

The following dialog box appears:

Clicking the arrow icon of the drop-down box or pressing the F3 key brings up a lens with all transactions of the given security. Select the transaction to cancel in this lens. (The input field itself currently has no function, don't enter any number here.)

The result is a number of new and as yet unposted transactions. The first is a cancel transaction to "negate" the selected transaction itself; you see that this is a cancel transaction by the display of CANCELATION left to the ID field.

The second new, as yet unposted transaction is a copy of the canceled transaction. The idea behind this: Probably you cancel a transaction because it's wrong somehow, e.g. with a wrong amount, not because you want to delete it outright without any replacement. So now you can modify this copy which is easier than re-starting from zero.

If you do want to "delete" a transaction i.e. cancel it without replacing it somehow just delete this copy.

Any transactions for the security that took place after the one to cancel will be canceled and immediately reposted: For each such transaction you get two new unposted transactions, one to cancel and one to repost. (Do not change those repost transactions in any way.)

The idea behind this: The canceled transaction can influence transactions that follow it, especially as far as FA amounts for profit/loss are concerned. Instead of generating FA journal entries that merely book resulting differences which would be hard to understand later it's better to cancel and repost everything.

A few more points about canceling: As an exception to the usual rules about unposted transactions normal users are not allowed to delete a cancel transaction; once they created one they have to go through and post it later. An attempt to delete such a transaction leads to the following error dialog box:

If you have at least Superuser privileges (and thus know all the details, it's assumed) you get to decide about the fate of the transactions that follow the transaction to cancel, because canceling and reposting those is unnecessary in certain circumstances:

And finally: Nothing is really canceled until the transactions are posted.

# Working With Prerecorded Transactions

Moneysoft has a pool for so-called prerecorded transactions. These are transactions that most probably will be have to be posted with a known trade date in the future, typically transactions for earnings like regular dividends. Prerecorded transactions "wait" in their own pool separate from the pool of normal unposted transactions as managed by the program Record Transactions until their time has come.

Prerecorded transactions are mostly generated automatically based on dividend data (see the chapter about program Manage Dividends), but there is also the program Prerecord Transactions in program group Transactions where you can create such transactions manually if needed.

Prerecorded transactions can't be posted directly: You have to "take them over" in program Record Transactions i.e. let the system create unposted transactions based on the prerecorded data and delete the prerecorded transactions afterwards.

There is the following mechanisms: If the program detects matching prerecorded transactions after you specify security and transaction code it offers you take them over by displaying the following dialog box:

After answering with "Yes" you get a lens listing all matching prerecorded transactions. After selecting one in that lens you get a second dialog box:

Answering with "Yes" on this box and then "Yes" again on the main dialog of the program gets you a corresponding unposted transaction, with the prerecorded transaction deleted.

# Net, Gross, Cost, Book and Market Values in Valor

# Introduction

When working with Valor you work daily with several "values": cost value (or acquisition value), book value, and market value. If e.g. you have a client that works with mark to market and do your year-end valuation as at 31.12. using market values, at the end of that day, for that client, book value and market value are identical.

This chapter tries to make clear which value is which, and show how Moneysoft and Valor in particular deal with the different values.

On the most basic level, Moneysoft works with net values and book values respectively. Book value is fundamental for calculating unrealized profit/loss as the difference between book and market value. Book value is influenced by normal transactions, but also by valuation transactions at the end of a period.

# Foreign Exchange Contracts

In Moneysoft you manage foreign exchange contracts (abbreviated FEC) in Valor, in contrast to cash transactions which you manage in Final.

Here are the details how a single FEC is processed:

When opening a FEC you enter a buy amount, the receivable, and a sell amount, the payable. As this transaction will take place in the future it does yet belong to the balance sheet; that's why for FECs an account group separate from profit and loss accounts and balance sheet accounts is used, called out of balance (German: Ausserbilanz).

The journal entries for those out of balance accounts resulting from the opening of the FEC record exactly the buy and sell amounts and remain unvalued until the FEC is liquidated. For FECs you will work with 3 out of balance accounts per currency, one for the receivables, one for the payables, and one for the valuations.

If you value FECs (see chapter Valuations) Moneysoft uses the forward rates to calculate the difference to the buy amount (the unrealized profit/loss).

It won't post this to the accounts for the receivables and the payables, but to the separate valuation account: A valuation will result in a journal entry for the valuation account; its amount will be the difference between the current profit/loss and the profit/loss at the time of the last valuation, to bring the balance of the valuation account to the current profit/loss.

To value FECs forward exchange rates are needed for each occuring currency, possibly client-specific. Moneysoft will calculate missing rates by using interpolation. Example: The forward rates 1W, 1M, 2M, 3M and 1Y are present, but a rate 4 months out is needed. Moneysoft will take the 3M and 1Y rates and interpolate a rate 4 months in the future.

Note

If you work with spot deals when rolling over FECs it's recommended to treat those deals technically like FECs although they aren't of course. This is simply the easiest way to assure that the same accounts are used for profit and loss and keep the whole accounting consistent.

# Recording New FECs

To record a new FEC you use the program Record Forward Exchange Contracts in program group Transactions:

From a technical point of view a FEC in Valor consists of a special security that only ever gets transactions related to the FEC. Each FEC get its own such security. You specify everything needed to create it in the frame Basic data.

The program assumes the following convention about keys: All FECs for the same currency pair in the same direction share a common securities number but differ in the postfix.

The lens to the right lists all existing FECs. You can start to record another FEC based on an existing one by selecting it in this lens: First, the securities number will be set, and after you enter a postfix or enter empty on that field to get the first available postfix all applicable data from the existing FEC will be copied into the fields of the dialog.

The program will create an unposted transaction for starting the FEC from the values in frame Transaction info. You can ask for a second unposted transaction for the liquidation to be created right away as well by filling the fields in frame Liquidation. The program calculates and proposes a voucher date as maturity date minus 2 working days, as per convention for FECs:

(The resulting liquidation transaction gets the maturity date of the FEC as value date, voucher date as entered here, and trade date identical to voucher date.)

After you confirm with "Yes" the program creates the security behind the FEC, including client-specific data, an opening transaction and possibly a liquidation transaction, both unposted.

Important

The liquidation transaction will stay unposted at least until its trade date is reached; the program Post Transactions will only process liquidation transactions if trade date <= current date. (Other types of transactions in the future are posted right away, however.)

If you have to record a FEC for a new currency pair you have to complete its basic data yourself; the program will just propose default values for Type and Unit:

In frame Transaction information enter the currencies and the amounts. The program will calculate the exchange rate:

If you already defined an account assignment for the currency pair of the FEC earlier you can select it to fill the account number fields in the frame Account info with the correct values. The fields show you which accounts are needed. You will need a transitory account if both currencies differ from the base currency of the client.

In the case of a FEC for a new currency pair you probably won't have an assignment yet however; in this case use a new number on the field Account assignment, name the assignment on the following field, enter all the account numbers, and the program will create a new account assignment accordingly. (The numbers for the current accounts will default to values defined in program Manage Current Accounts in program group Basic Data.)

You post transactions from FECs by using the specification Post FXF contracts in program Post Transactions in program group Transactions.

Note

The opening of a FEC will generate journal entries for the "receivable" and "payable" ouf of balance accounts. Profit and loss handling happens at liquidation.

# Valuation

You have to actively value FECs if you want them to appear on reports printed by program Print Portfolio, if you need profit/loss data for a given key date or if you need profit data for a performance calculation.

Important

If there is no valuation between issue date and liquidation date, the system will automatically create one valuation record for the FEC in the database for each working day at the liquidation date at the latest.

# Value Open FECs

The program that values FECs is Value Forward Exchange Contracts in program group Valuations:

This is how to read the resulting report: Our Sale and Our Buy show the contracted amounts. Current Value is as at valuation date, using Current Forward Rate. The difference between current value and the buy amount is unrealized P/L Currency.

# Checking FEC Valuations

As already explained in the previous chapter, the program Value Foreign Exchange Contracts in program group Valuations stores all calculated valuations into a database.

If you merely want to check those valuations later, there is a second program Print Foreign Exchange Contracts in program group Transactions.that does neither calculate nor store anything, but simply brings the current content of that database to paper: It prints lists of all stored valuations within a specified date period.

# Liquidation

If you opted not to let program Record Foreign Exchange Contracts generate the liquidation transactions together with the opening transactions, you have to generate them later by using the program Liquidate Forward Exchange Contracts in program group Transactions.

The principle is the same however: The program will create unposted transactions for each FEC to liquidate according to the date values specified in the dialog which you will have to post later.

The program only works on FECs within the current client.

Enter the selection values as follows:

Value date: The maturity date of the FECs to liquidate

Trade date: The voucher date of the FECs to liquidate, as per convention maturity date minus 2 working days

Voucher date: Identical to trade date

Transaction code: 40

Transaction description: Accept the default Close FX Forward

# Cancel FECs

You cancel FECs in program Record Foreign Exchange Contracts in program group Transactions. You have two options that you choose by using the corresponding button:

Cancel FEC: All transactions, FA journal entries and valuations belonging to the FEC are canceled. After that you can't use the security that represents the FEC anymore; for a new (and corrected) FEC you have to create a new security i.e. use a new postfix.

Cancel FEC Liquidation: Only the transactions, FA journal entries and valuations belonging to the liquidation are canceled. The FEC itself i.e. everything belonging to its opening stays as it is, and you can liquidate it again after this.

A click on either button will bring up a lens with available FECs where you select the one to cancel.

Canceling here works like everything else in Valor: It's only really canceled after you post the resulting unposted transactions.
