# Administration

# Creating New Clients

A new client must be created in Final first, then in Valor, and finally in Emcal. If this order is not respected, some basic data may be missing, because the program for creating Valor clients checks data in the corresponding Final client - if present.

# New Final Client

You create a new Final client with the help of the program Manage Clients in program group Clients.

You have to use a new number which you cannot change once the client is created. Names and short names are free to modify however even for existing clients.

For creating a client you have to specify a template client. You can take any existing client; best of course is a client that is as similar as possible to the new client. You can copy basic data from the template client by using the options Copy informations?, Copy chart of accounts? and Copy reports?.

If the numbers in the chart of accounts must obey a certain "pattern" you can specify this pattern on the input field Account number pattern. Whenever you enter an account number in any of the Final programs the system will check the number against this pattern and will complete the number if there is a unique match with an existing account.

Possible characters for building patters are:

Start of fiscal year defines the earliest possible date for a journal entry and the fiscal year itself: The year component of this date is the fiscal year; e.g. with a start date of "01.10.2015" you get 2015 as the fiscal year. For overlong fiscal "years" that are longer than a calendar year you don't have to configure anything special; just set/leave the start of the fiscal year accordingly.

You must specify the Base currency already when creating the client, and you cannot change it anymore later.

The list of VAT codes that the client will use depends on the Domicile that you specify. If there are no VAT codes listed for the domicile in the master code list the VAT codes of the template client will be copied.

You have to declare clients that are subject to VAT as such by setting the Subject to VAT? checkbox. The programs will then fill VAT info when you create journal entries.

You can define a Print language for a client. You refer to this language by using the language selection mode Client language (part of the Standard options of every program that prints.)

On Voucher number you can set the way voucher numbers are assigned when creating journal entries. A new journal entry will get the value of Last voucher number plus 1. Thus if you want the voucher numbers for your new client to start e.g. with 1000, enter 999 on this field when creating the client.

# New Valor Client

You create a new Valor client with the help of the program Manage Clients in program group Clients.

Use the same number for the Valor client as for the Final client created beforehand. You cannot change this number once the client is created. Names and short names are free to modify however even for existing clients.

For creating a client you have to specify a template client. You can take any existing client for this. For the correct copying of basic data from the template client to the new one do not check Copy client-specific data?. Do check Copy current accounts?.

Start of fiscal year defines the date range for transactions: The date is the earliest possible date for a transaction.

You have to specify a Domicile; among other things, the domicile selects refund rates when recording transactions.

The National currency will usually be CHF. The system will then use CHF for calculating cross exchange rates whenever direct cross rates are not available. You have to import exchange rates with CHF as base currency for this to work. If you want to work with exchange rates with USD as their base currency instead you can use USD as national currency instead of CHF.

You can define a Sector code scheme to be used as standard for this client. You use sector code schemes e.g. in certain reports or for sector-wise performance calculation. For this to work you have to assign a sector code from this scheme to each security that the client will hold.

You can configure the client to use a certain transaction set. With the field Transaction set empty the system will use the standard transaction set.

Check the option Calculate FIFO? if the realizations for all sell and buy transactions in this client should be posted according to the FIFO ("first in, first out") principle.

You can define a Print language for a client. You refer to this language by using the language selection mode Client language (part of the Standard options of every program that prints.)

In the frame Accounting you configure the interface from Valor to Final, how voucher numbers are assigned and which currency will be the base currency.

# New Emcal Client

You create a new Emcal client with the help of the program Manage Clients in program group Clients.

Use the same number for the Emcal client as for the Final and Valor clients created beforehand. You cannot change this number once the client is created. Names and short names are free to modify however even for existing clients.

For creating a client you have to specify a template client. You can take any existing client for this. You don't need to check Copy definitions?.

Do set Currency and Price frequency.

# Next Steps after Creating the Clients

A new client starts with copies of document contents and configurations from the template client used. You can then complete and/or modify this data for this particular client. It's thus important that you choose a suitable template client for your new client, and likewise important that the data in the template clients is correct and appropiate. Otherwise you waste your time adjusting the same things over and over again for each new client.

# Main Configuration Data with Regular Modifications

# Documents

A document within the domain of Moneysoft configuration data is a small table that is mostly static, with modifications only from time to time or never at all. (Such a document has nothing to do with text documents as written e.g. with Microsoft Word.)

Some documents are visible to users, e.g. as lists of possible codes in lenses on input fields. Other documents are invisible in the background, containing parameters that are consulted by programs as they run.

There is a system program called Document Editor that can be used to modify the content of a document. It's available through the function Document Editor in menu Office.

Documents are not stored as separate files, but as parts of files like e.g D000Vrdk.DAT. To modify a document after starting the editor directly you have to know the file and the name of the document.

For a document that is visible on an input field in some program, there is a much easier way however: Start the program, navigate the cursor to the field and press F2 (shortcut for function Associated record in menu Help.)

There is a large number of documents in Moneysoft, with many different kind of data and roles. It would go much too far to describe them all here; if in doubt, please consult Megos.

# Deposits

The list of deposits is a central peace of Valor configuration data. It's important to define the list correctly already early on because many other kind of data like the client-specific data for securities will depend on them. Deposit IDs with up to 5 characters are part of the primary key in a number of important data tables like the one for transactions, and once used you cannot change a deposit number anymore.

Beside its ID a deposit also has a name. You can freely change the name. So, if you have a deposit with ID "UBS" and name "UBS" as well, you could change the name to "UBS Geneva Branch" later without problems, but not the ID to "UBSGV".

The list of deposits is client-specific. If you create a new Valor client, its initial deposit lists comes from the template client. After this you can change it independently. The list is stored in document Depotstelle in file DxxxMadk.DAT (with "xxx" the three-digit client number).

# Current Accounts

Curret accounts can be used to link transactions to balance sheet accounts in Final. The program Record Transactions will use data for current accounts to propose default values for the FA accounts to use. You can e.g. configure that all income goes to a CHF account at UBS but that money from buy and sell transactions goes to accounts with the currency of issue of the security in question.

New clients start with a copy of the current account list of the template client. There are a number of reasons to modify this list:

  • The client switches to another bank
  • The client restructures his or her deposits
  • The client does not strictly use accounts in transaction currency

In program Manage Current Accounts in program group Basic Data you can use 3 different criteria Deposit, Type of transaction and Security type to define when a given current account should be taken.

By using these flexible criteria you can achieve a very high degree of automatization in program Record Transactions which will automatically default to all the right accounts for all kinds of transactions. It's still possible to manually change accounts in special cases of course.

# Account Assignments

An account assignment defines which FA accounts are used for posting the various amounts resulting from a transaction. Like with other configuration data a new client starts with a copy of the account assignments defined in the template client used, but later on you can modify them independently for each client.

You find them in document Kontierung in file DxxxMadk.DAT, with "xxx" standing for the client number.

You find general info about how to modify a document such as this one in chapter 2.1.

A possibility to use the F2 shortcut into the document editor with this document is the field Account assignment in program Manage Client-Specific Data in program group Basic Data.

Usually you work with one account assignment per security account, so that cost value and profit/loss can go to different FA accounts.

# Securities Traders and Stamp Duty

# Introduction

All the following refers to the circular Kreisschreiben Nr. 12 of the Swiss Federal Tax Administration from March 10, 2011. It only applies to clients which are securities traders ("Effektenhändler") in the sense of this circular.

If a securities trader registers with his Swiss trading partner, i.e. the bank, that bank will not charge stamp duty for transactions because it is the obligation of the securities trader calculate and pay the stamp duty. Some banks, howewer, offer the service of calculation and payment. In this case, the .

Somebody with securities trader status has the following obligations:
a) To collect stamp duty for any asset management activity for Swiss parties without involvement of a bank
b) To collect stamp duty for any direct business with foreign brokers and banks (delivery versus payment to the Swiss correspondent bank)
c) To document any stamp duty that was directly collected by a Swiss bank
d) To calculate and report a differential payment each quarter

It's possible that this differential payment amounts to 0: That's the case when all transactions of a given quarter were of type c) and correctly settled in regard to stamp duty. The report then documents all the transactions subject to stamp duty and states the Swiss party collecting the duty for each of them.

The Valor program Print Transfer Tax Register in program group Reports is able to produce such a report. Based on transaction codes it only selects transactions that involve stamp duty. (Currently, the system has to rely on transaction codes because it does not know whether the traded securities as such are subject to stamp duty or not.)

# Processing Banks for Transactions

When recording transactions with program Record Transactions in program group Transactions the field Processing bank must be filled for each buy and sell transaction.

If that bank collects stamp duty the amount must be stated in field Stamp duty.

If however the securities trader is responsible for the stamp duty a second transaction must be recorded, with 22 as transaction code, and with the stamp duty amount as settlement amount. (The user calculates the amount himself, there is no automatism.)

# Report Generation and Corrections

The program Print Transfer Tax Register in program group Reports is used to generate the report. The report prints its results into a Excel file. Not everything is fully automated; you have to check that Excel file and manually correct some things in the file as follows:

  • If a transaction uses one of the transaction codes configured for stamp duty according to the previous chapter 3.2.3, but the traded security itself is exempted from stamp duty, that transaction will be in the report erroneously: Delete all lines for such transactions from the report (e.g. if used 11 for buy of money market call money).
  • There is no duty necessary on particularly instruments, which contain a choice/option. E.g. Warrants, Options, Futures, Calls und Puts and the like. Short term investments such as fiduciary accoutns, Money Market Funds, Credits/Loans are exempt of duty, too.
  • With transactions that are processed through a foreign bank or broker there is no entry into the comments column: (Bemerkungen/Comments) but you must enter the whole gross turnover in the relevant column for domestic/foreign security (in-/ausl. Titel). This can be done by Moving the amount from the existing report cell to the relevant cell domestic/foreign for the same line.
  • One needs to amend the calculation totals by enterring a formula in int the sum/total cells: Domestic Total * 0.075%: Foreign Total *0.15%. The page totals and summ cells need a fine tuning, too. The overal total of the domestic securities plus the overall total of the foreign securities x 0.15% have to match with the balance of the account Stamp Duty.

There is special care to be taken for the column “Comments”.

With transactions that are processed with a Swiss bank, the short key for that particular bank will be entered manually into the EXCEL, e.g. CSZRH, UBSZRH. Both amounts are to be entered as „nicht pflichtig“ (no duty) as the amounts are transferred by the processing bank.

Big differences are to be analysed and are to be corrected in either adding Transactions of 22, correcting masterdata of brokers, adding execution brokers to the transactions or other measurments.

# Specifications

Reports as they are printed by Final and Valor print programs typically have many parameters that influence exactly what is printed, and how. A set of parameter values is called specification. A specification comprises a name, standard options, possibly special options, and possibly selections.

Each print program has its own list of specifications. Specifications are client-independent, or in other words valid for all clients.

To create a new specification or to modify an existing one you use Edit specification on the field Mode in the print program:

If you enter the name of an existing specification on the field Specification while in this mode, you can modify it. To create a new specification, enter a new name on that field, which will switch the program into Create mode.

You can specify or change the main options and the descriptions for the specification on the dialog behind the button Standard Options….

The language to use for the report, the sort order for the data to print and the printer to use are among those main options:

The button Special Options… leads to a dialog with more, this time program-specific options. The following screenshot shows a typical special options dialog. Note that sometimes, depending on the print program, standard and special options interact in some way, e.g. there is a special option Print subtotals? to activate subtotals, but the sort order as defined in the standard options decides what is totalled.

Part of most specifications are selections which you enter on a dialog behind the button Selections…. Selections are program-specific. With them you can specify conditions that data records have to meet in order to be "selected" and printed.

# FA Securities

# Creating FA Securities

You can include balance values from FA accounts in Valor reports like the one generated by program Print Portfolio as if they were "normal" securities with a balance based on recorded transactions, with the help of so-called FA securities.

A security is treated as such if you specify an account group for it with the program Manage Client-Specific Data, on the fields in the frame Account group of its main dialog.

Because this artificial security has of course no natural ID like an IBAN or a Valor number use an internal number like e.g. "ZZBAC000000x" for a current account.

The account balances are computed as needed at the time of the report; there is no need to export or prepare anything beforehand.

On the field Group you specify the account group and thus the accounts that will go into this FA security. The next chapter 5.2 explains how to create or modify account groups.

On the field Type you declare the security to be either Capital or Earnings which plays a role when calculating performance.

# Account Groups

Account groups are defined in document KontoGruppe in file DxxxMadk.DAT. The easiest way to modify the document is by using the F2 shortcut for entering the program Document Editor with the cursor on field Group in program Manage Client-Specific Data.

Each client has its own list of account groups.

For a group comprising a single account or a single account range create one entry in this document, giving the desired group number on the field Group.

For a group comprising several account ranges create several entries, with identical group number on the field Group, and with the boolean Addieren? set to Y.

When creating entries, give your entries appropriate "artificial" key values on the field Key that are required for technical reasons and do not define the groups themselves.

# Peformance Calculations

# Calculate Performance Data

The program Calculate Performance Data in program group Reports is the program that calculates performance data, prints it in a simple way and stores it permanently into a database.

As a print program it works with specifications like explained in chapter 4, but with the difference that here, as a rare exception, the specifications are per client. Furthermore there is the option Calculate all specifications? that processes all specifications for the active client in one go, with an arbitrary specification with that option set used to start the process.

For each specification performance is calculated for aggregates, the so-called total types, and depending on the options set for securities and/or client-specific securities.

You can label the data calculated by a specification with a so-called selection method; specify its code on the field Selection method if wanted. The available selection methods are contained in document PerfSelektion in file DxxxMadk.DAT; use the F2 shortcut with the cursor on the field to enter Document Editor if you want to define new selection methods.

So by using selections on the dialog behind the button Selections… and labelling the corresponding calculated data with the help of selection methods you can handle performance data for several sub-portfolios; use the names of the selection methods to keep the overview which is which.

The longest possible period for a performance is 1 year. Performance data is stored per key date, per period and per calculation method.

# View Performance Data

Calculated performance data can be viewed on screen in program Manage Performance Data in program group Reports.

Choose a selection method and a period to view data. There is data per period and per security, for all available total types, performances for a day, for a month, for a quarter, for a year and since inception.

All these performances are calculated using several methods: TTWR, MWR, Modified Dietz, Money multiplier and IRR annualized.

# Print Performance Data

You print performance data or export it to Excel with the program Print Performance Data in program group Reports.

# VAT

If you create a new Final client its list of VAT codes will be generated based on the domicile given and the VAT code master document. If you specify a domicile that is missing in the master document the VAT code list of the template client will be used.

The master document UmsSteMas with VAT codes per domicile is in file D000Fidk.DAT.

The boolean option VAT included? sagt ob der Betrag i.d.R. inklusiv oder exklusiv MWST eingegeben wird, dementsprechend wird der Net amount und der Taxable amount gerechnet.

The boolean option Net? sagt ob die Netto- oder Brutto Abrechnungsmethode angewendet wird. Wird Brutto-Methode angewendet, wird beim Verbuchen keine MWST Buchung erstellt. Diese wird erstellt with the program Settle Value Added Tax in program group Journal Entries.

You can set a default VAT code to be used for journal entries involving a given account in program Manage Chart of Accounts in program group Chart of Accounts, on the field Tax code. There is also an option This tax code exclusively?; if set, no other VAT code will be allowed for this account.

With the option All entries with tax? you can force all journal entries for the account to have VAT.

Example of a net settlement, excluding VAT, with 2 journal entries:

100000/340000: CHF 925.93
100000/220000: CHF 74.07

Example of a gross settlement, including VAT:
First journal entry posted:

100000/340000: CHF 1’000

Second journal entry generated by program Settle Value Added Tax:

340000/220000: CHF 74.07
